## Info Pages (iPages)

### What They Are

Standalone pages with custom HTML/JS content, accessible via the CRM sidebar menu or direct URL. Think of them as wiki pages that can also run JavaScript.

### Table: `app_ext_ipages`

```
+-------------+--------------+------+-----+---------+----------------+
| Field       | Type         | Null | Key | Default | Extra          |
+-------------+--------------+------+-----+---------+----------------+
| id          | int(11)      | NO   | PRI | NULL    | auto_increment |
| parent_id   | int(11)      | NO   |     | 0       | (menu parent)  |
| name        | varchar(255) | NO   |     |         | (display name) |
| short_name  | varchar(64)  | NO   |     |         | (menu label)   |
| menu_icon   | varchar(64)  | NO   |     |         | (FA icon class)|
| icon_color  | varchar(7)   | NO   |     |         | (hex color)    |
| bg_color    | varchar(7)   | NO   |     |         | (hex color)    |
| description | longtext     | YES  |     | NULL    | (CKEditor HTML)|
| html_code   | text         | NO   |     |         | (raw HTML/JS)  |
| users_groups| text         | NO   |     |         | (CSV group IDs)|
| assigned_to | text         | NO   |     |         | (CSV user IDs) |
| sort_order  | int(11)      | NO   |     | 0       |                |
| is_menu     | tinyint(1)   | NO   |     | 0       | (menu folder?) |
| attachments | text         | NO   |     |         | (file list)    |
+-------------+--------------+------+-----+---------+----------------+
```

**Key columns:**

- `description` - Rich text content edited via CKEditor in the UI. Rendered first, above html_code.
- `html_code` - Raw HTML/CSS/JavaScript. NOT processed by PHP -- output directly via `echo`. This is where custom dashboards live.
- `users_groups` - CSV of access group IDs (e.g., `0,4,5,6`). Group 0 = admin.
- `assigned_to` - CSV of individual user IDs who can see the page.
- `is_menu` - If 1, this page acts as a **menu folder** (parent for other pages). If 0, it is a regular content page.
- `parent_id` - Points to another iPage's `id` where `is_menu=1` to nest under it in the sidebar.
- `short_name` - Used as the menu label when present (falls back to `name`). Also used for URL slug-style references.

### URL Pattern

```
index.php?module=ext/ipages/view&id=<ipage_id>
```

Generated by: `url_for('ext/ipages/view', 'id=' . $id)`

### How Rendering Works

**File: `crm/plugins/ext/modules/ipages/views/view.php`**

Render order:
1. Page title: `<h3 class="page-title">{name}</h3>`
2. Description (CKEditor content): `ipages::prepare_attachments_in_text($ipage['description'], $ipage['attachments'])`
3. Attachments output (if any)
4. HTML/JS code: `echo $ipage['html_code']` -- **raw output, no PHP processing**

**Critical: `html_code` is echoed directly as HTML. It does NOT execute PHP.** You cannot put `<?php ?>` tags in html_code. For server-side logic, you must use one of the external loading patterns.

### Access Control

**File: `crm/plugins/ext/modules/ipages/actions/view.php`**

```sql
SELECT * FROM app_ext_ipages
WHERE id = :id
AND (
    FIND_IN_SET(:user_group_id, users_groups)
    OR FIND_IN_SET(:user_id, assigned_to)
);
```

If no match, redirects to `dashboard/page_not_found`. Admin (group_id=0) must still be in the `users_groups` CSV.

### Menu System

**File: `crm/plugins/ext/classes/ipages.php` -- `build_menu()`**

- iPages appear in the sidebar menu automatically if the user's group/ID is in `users_groups`/`assigned_to`.
- Pages with `is_menu=1` become **collapsible menu folders** with nested children.
- Pages assigned to an entity menu (via `app_entities_menu.pages_list`) are excluded from the main sidebar to avoid duplicates.
- Menu uses `short_name` if set, otherwise `name`.

### iPage Admin Access

Controlled by two config constants:
- `CFG_IPAGES_ACCESS_TO_USERS` - CSV of user IDs who can manage iPages
- `CFG_IPAGES_ACCESS_TO_USERS_GROUP` - CSV of group IDs who can manage iPages

Users with access see the "Info Pages" config link in the menu. Admin (group_id=0) bypasses this if `group_id>0` check fails.

## Three Patterns for Custom iPage Content

### Pattern 1: Self-Contained (inline CSS+HTML+JS)

**Used by: Pipeline (id=8), Claude Chat (id=1), Quick Estimate (id=7)**

Everything lives in `html_code`. JS fetches data from the CRM API or plugin endpoints.

```html
<style>
  .my-widget { ... }
</style>
<div id="my-widget">Loading...</div>
<script>
  $.get("plugins/claude/my_endpoint.php", function(data) {
    // render data into #my-widget
  }, "json");
</script>
```

**Pros:** Single source of truth, easy to update via SQL.
**Cons:** Hard to edit large codebases in a textarea. No PHP execution.

### Pattern 2: External JS File

**Used by: Mission Control (id=6)**

html_code contains CSS + HTML skeleton + a `<script src>` tag:

```html
<style>/* all CSS here */</style>
<div id="mission-control">
  <h2>Mission Control</h2>
  <div id="mc-tree">Loading...</div>
</div>
<script src="plugins/claude/mc3.js"></script>
```

**Pros:** JS file is a real file you can edit with normal tools. CSS stays in html_code for easy theming.
**Cons:** Still no server-side PHP in the page itself. JS must call API endpoints for data.

### Pattern 3: Iframe to PHP

**Used by: Analytics Dashboard (id=5)**

html_code is just an iframe pointing to a PHP file:

```html
<iframe src="plugins/claude/analytics_briefing.php"
        style="width:100%;min-height:600px;border:none;"></iframe>
```

**Pros:** Full PHP execution. Can query the database, include libraries, etc.
**Cons:** Runs in an iframe (separate DOM, no access to parent CRM JS/CSS). Cross-frame communication requires postMessage.

### Pattern 4: AJAX to PHP Endpoint

**Used by: Websites Dashboard (id=3)**

html_code contains the full UI skeleton + JS that calls a PHP endpoint via AJAX:

```html
<style>/* all CSS */</style>
<div class="wd-container">
  <div id="statsCards"><div class="wd-loading">Loading...</div></div>
  <!-- more skeleton HTML -->
</div>
<script>
  $.get("plugins/claude/websites_dashboard.php", function(data) {
    // populate the skeleton with data from PHP
  }, "json");
</script>
```

The PHP endpoint (`websites_dashboard.php`) returns JSON. The JS renders it into the DOM.

**Pros:** Full PHP on the backend. Frontend runs in the CRM's DOM (can use jQuery, CRM styles, etc). Best of both worlds.
**Cons:** Requires both a DB entry (html_code) and a PHP file.

**This is the recommended pattern for new custom pages.**

## Main Dashboard

### How It Works

**Entry point:** `index.php?module=dashboard`

**Routing:**
1. `crm/modules/dashboard/actions/dashboard.php` - handles AJAX actions (save, sort, etc.)
2. `crm/modules/dashboard/views/dashboard.php` - renders the dashboard view

**Plugin override:** The dashboard view first checks for plugin dashboards:
```php
foreach(explode(',', AVAILABLE_PLUGINS) as $plugin) {
    if (is_file('plugins/' . $plugin . '/includes/dashboard.php')) {
        require('plugins/' . $plugin . '/includes/dashboard.php');
        $is_plugin_dashboard = true;
    }
}
if (!$is_plugin_dashboard) {
    require(component_path('dashboard/dashboard_default'));
}
```

If a plugin provides `includes/dashboard.php`, it **replaces the entire default dashboard**. Currently no plugin does this in our install.

### Default Dashboard Render Order

**File: `crm/modules/dashboard/components/dashboard_default.php`**

1. **Reports Groups Tabs** - `reports_groups::render_dashboard_tabs()` (tabs at top, from `app_reports_groups`)
2. **Dashboard Pages - Info Blocks** - `$page->render_info_blocks()` (colored stat cards from `app_dashboard_pages`)
3. **Dashboard Pages - Info Pages** - `$page->render_info_pages()` (text blocks from `app_dashboard_pages`)
4. **Report Counters** - `$reports_counter->render()` (badge-style counts)
5. **Sections** - from `app_reports_sections` (1 or 2 column layouts with assigned reports)
6. **Standard Reports** - user's own reports where `in_dashboard=1`
7. **Report Pages** - from `app_ext_report_page` where `in_dashboard=1`
8. **Common Reports** - shared reports visible to the user's group

### User Configuration

Users configure their dashboard via a modal (`dashboard/configure`):
- **Standard Reports tab** - drag reports between "On Dashboard" and "My Reports" lists
- **Sections tab** - create 1 or 2 column sections, assign reports to left/right slots
- **Counters tab** - drag reports to show as count badges
- **Header tab** - drag reports to show in the top header bar
- **Common Reports tab** - toggle visibility of shared reports

### Create a New iPage

```sql
INSERT INTO app_ext_ipages (

    description, html_code, users_groups, assigned_to,

    '',                         -- icon_color (hex or empty)
    '',                         -- bg_color (hex or empty)
    NULL,                       -- description (CKEditor HTML, or NULL)
    '<div id="my-page">Loading...</div>\n<script>\n// your JS here\n</script>',
    '0,4,5,6',                  -- users_groups (admin + other groups)
```

## Patterns & Techniques

### Recommended: New Custom Dashboard Page

1. Create a PHP endpoint at `crm/plugins/claude/my_feature.php` that returns JSON
2. Create an iPage with html_code containing CSS + HTML skeleton + AJAX loader
3. The AJAX loader calls your PHP endpoint and renders the response

```sql
INSERT INTO app_ext_ipages (
    parent_id, name, short_name, menu_icon, users_groups, sort_order, is_menu,
    html_code, description, icon_color, bg_color, assigned_to, attachments
) VALUES (
    0, 'Feature Name', 'feature', 'fa-star', '0,4,5,6', 10, 0,
    '<style>\n#feature-page { padding: 20px; }\n</style>\n<div id="feature-page"><div class="loading">Loading...</div></div>\n<script>\n$.getJSON("plugins/claude/my_feature.php", function(data) {\n  // render\n});\n</script>',
    NULL, '', '', '', ''
);
```

### jQuery is Available

The CRM loads jQuery globally. All iPage html_code runs in the CRM's DOM context, so you can use:
- `$()` / `jQuery()`
- `$.get()`, `$.post()`, `$.ajax()`, `$.getJSON()`
- `url_for()` (CRM's JS helper for generating URLs -- available in the global scope)
- Bootstrap 3 components (modals, dropdowns, tabs, etc.)

### Accessing CRM Data from iPage JS

Use the CRM REST API from JavaScript:
```javascript
$.ajax({
    url: '/crm/api/rest.php',
    method: 'GET',
    data: {
        username: 'claude',
        password: 'badass',
        key: 'dZrvuC3Q1A8Cv82X1dQsugRiCpmU3FSMHn9BFWlY',
        action: 'select',
        entity_id: 25,  // Leads
        limit: 10
    },
    success: function(response) { /* ... */ }
});
```

Or call a custom plugin PHP endpoint that handles the DB queries server-side.

### Updating iPage Content Programmatically

Since html_code is just a TEXT column, Claude can update it via SQL:
```sql
UPDATE app_ext_ipages SET html_code = :new_content WHERE id = :id;
```

This is how we maintain pages like Pipeline and Mission Control -- the PHP/JS logic lives in files, but the CSS and HTML skeleton live in the database.

### Dashboard Plugin Override

To completely replace the default dashboard, create `crm/plugins/<name>/includes/dashboard.php`. If that file exists and the plugin is in `AVAILABLE_PLUGINS`, it replaces the default dashboard for all users. Use with caution.

### Key File Paths

| File | Purpose |
|------|---------|
| `crm/plugins/ext/modules/ipages/actions/view.php` | iPage view action (access check, attachment handling) |
| `crm/plugins/ext/modules/ipages/views/view.php` | iPage view template (renders description + html_code) |
| `crm/plugins/ext/modules/ipages/actions/configuration.php` | iPage CRUD (save, delete) |
| `crm/plugins/ext/modules/ipages/views/configuration_form.php` | iPage edit form (3 tabs: General, Access, Code) |
| `crm/plugins/ext/modules/ipages/components/menu.php` | Sidebar menu builder |
| `crm/plugins/ext/classes/ipages.php` | ipages class (build_menu, get_pages, access helpers) |
| `crm/includes/classes/pages/dashboard_pages.php` | Dashboard info blocks/pages renderer |
| `crm/modules/dashboard/components/dashboard_default.php` | Default dashboard layout orchestrator |
| `crm/modules/dashboard/components/sections.php` | Dashboard sections renderer (1/2 column layouts) |
| `crm/modules/dashboard/components/sections_reports.php` | Report type dispatcher for sections |
| `crm/modules/dashboard/components/report_page.php` | Report page blocks on dashboard |
| `crm/modules/dashboard/components/common_reports.php` | Shared/common reports renderer |
| `crm/modules/dashboard/views/configure.php` | Dashboard configuration modal |
| `crm/modules/dashboard/actions/dashboard.php` | Dashboard AJAX actions (sort, save, etc.) |

### Gotchas

1. **html_code is NOT PHP.** It is echoed raw. Use AJAX or iframes for server-side logic.
2. **Access requires explicit group assignment.** Even admin (group_id=0) must be listed in `users_groups`. The query uses `FIND_IN_SET(group_id, users_groups)` and 0 is a valid match for `'0,4,5,6'`.
3. **is_menu pages are folders, not content.** If `is_menu=1`, the page serves as a collapsible menu container. Its content still renders if you navigate to it directly, but its primary purpose is grouping child pages.
4. **Entity menu exclusion.** If an iPage is referenced in `app_entities_menu.pages_list`, it is excluded from the sidebar menu to avoid showing it twice. The `build_menu()` query filters these out.
5. **Dashboard sections are per-user.** The `created_by` column on `app_reports_sections` means each user has their own section layout. Sections with `reports_groups_id > 0` belong to a reports group tab.
6. **Reports on dashboard are per-user.** The `in_dashboard` flag is on the report row itself, and standard reports are filtered by `created_by`. Common reports are filtered by group access.
7. **description vs html_code.** Both render on the page. `description` renders first (with CKEditor formatting and attachment handling). `html_code` renders after. For custom dashboards, put everything in `html_code` and leave `description` NULL.
